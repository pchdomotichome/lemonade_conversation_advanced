# Analysis Report: lemonade_conversation_advanced — 'intent-failed' Diagnosis

## Executive Summary

The integration has **5 bugs** and **2 architectural issues** that collectively produce the `intent-failed` error with no HA logs. The root cause is that the streaming path silently drops tool calls, and when tools are needed for HA control, the LLM's tool-calling response is never executed.

---

## Issue 1: Streaming Path Drops Tool Calls (CRITICAL)

**File:** `conversation.py:427-556`

**Bug:** In `_async_handle_chat_log`, the streaming loop processes SSE deltas via `chat_log.async_add_delta_content_stream()`. Tool calls accumulated in `_parse_sse_stream` are yielded as `{"tool_calls": [...]}` dicts, but the loop only checks for `ToolResultContent` — it never triggers tool execution.

```python
# conversation.py:444-456 — the problem
has_tool_result = False
async for content in chat_log.async_add_delta_content_stream(
    self.entity_id,
    self._parse_sse_stream(resp, chat_log),
):
    if isinstance(content, conversation.ToolResultContent):
        has_tool_result = True

if has_tool_result:
    continue  # never reached — no tool execution happens
```

**Impact:** When the LLM returns tool calls (e.g., `turn_on_entity`, `get_entity_state`), the streaming path never executes them. The conversation entity returns empty/failed results, which HA interprets as `intent-failed`.

**Fix:** After the delta stream completes, detect if the chat_log contains `AssistantContent` with `tool_calls` and execute them via `chat_log.async_call_tool()`, following the pattern used in the non-streaming fallback path (lines 476-500).

## Issue 2: `voluptuous_openapi.convert()` on Already-Dict Parameters

**File:** `conversation.py:190-204`

**Bug:** `_get_tools` calls `convert(tool.parameters)` to convert voluptuous schemas to OpenAPI format. However, `llm.Tool.parameters` can be either a `vol.Schema` object OR a plain dict depending on the tool definition source. If `parameters` is already a dict, `convert()` may produce unexpected output or fail silently.

The HA core pattern is: tools define `parameters` as `vol.Schema`, and `convert()` is the correct approach. But the real issue is that this conversion can produce schemas with voluptuous-specific markers (like `vol.Optional` markers) that aren't valid OpenAPI, causing the LLM server to reject the tool definitions.

**Impact:** If tool schemas are malformed, the LLM server may return an error that gets caught but not logged (see Issue 4), producing `intent-failed`.

**Fix:** Add validation/fallback: if `convert()` fails or produces an invalid schema, fall back to a basic dict schema. Also verify that `tool.parameters` is a `vol.Schema` before calling `convert()`.

## Issue 3: RAG Index Created Without API Key

**File:** `__init__.py:96-99`

**Bug:**
```python
# __init__.py:96-99 — no api_key
cache_dir = f"{hass.config.config_dir}/lemonade_rag_cache"
rag_index = RAGIndex(cache_dir)  # ← api_key not passed
await rag_index.load()
hass.data[DOMAIN]["rag_index"] = rag_index
```

But in `conversation.py:407`, RAG is created WITH the API key:
```python
rag_index = RAGIndex(cache_dir, api_key)  # ← api_key passed
```

The `__init__.py` creates a RAG index without API key at setup, then `conversation.py` creates a *second* RAG index with the API key. This wastes resources and the `__init__.py` index is never used for actual queries.

**Impact:** If the RAG path in `conversation.py:411` tries to `refresh()` using the shared session (which may not have auth headers), embedding calls fail silently.

**Fix:** Either pass `api_key` in `__init__.py` when creating the RAG index, or remove the redundant creation in `__init__.py` since `conversation.py` creates its own.

## Issue 4: Silent Exception Swallowing

**File:** `conversation.py:412-413, 424-425`

**Bug:**
```python
except Exception:
    enable_rag = False  # line 413 — silent

except Exception:
    pass  # line 425 — completely silent
```

RAG-related exceptions are caught and discarded with no logging. If the embeddings endpoint fails (auth error, wrong model, timeout), the user sees `intent-failed` with zero diagnostic information.

**Impact:** This is why there are "no HA logs" — the errors are intentionally swallowed.

**Fix:** Log these exceptions at WARNING level: `_LOGGER.warning("RAG refresh/query failed: %s", err)`

## Issue 5: Non-Streaming `tool_args` May Be Dict Not String

**File:** `conversation.py:496`

**Bug:** In the non-streaming fallback path:
```python
await chat_log.async_call_tool(
    tool_name=tc["function"]["name"],
    tool_args=json.loads(tc["function"]["arguments"]),  # ← if arguments is already a dict, TypeError
    ...
)
```

Some LLM providers return `arguments` as a dict (already parsed), not a JSON string. `json.loads()` on a dict raises `TypeError`.

**Impact:** Non-streaming fallback fails, contributing to `intent-failed`.

**Fix:** Add type check: `args = tc["function"]["arguments"] if isinstance(tc["function"]["arguments"], dict) else json.loads(tc["function"]["arguments"])`

---

## Issue 6: Two Duplicate Conversation Entity Classes

**Files:** `conversation.py:47` (`LemonadeConversationEntity`) and `agent.py:103` (`LemonadeConversationEntity`)

**Issue:** Both files define a class named `LemonadeConversationEntity`. The `agent.py` version is the older, larger implementation that's NOT used by the active flow (it's not imported by `__init__.py` or `conversation.py`). But it exists in the codebase and could cause confusion.

The `agent.py` class doesn't inherit from `LemonadeBaseEntity` and uses a completely different architecture (MCP server calls, conversation history, etc.).

**Impact:** Not a direct bug, but dead code that complicates maintenance and could mask issues if someone accidentally imports from the wrong file.

**Fix:** Remove or clearly deprecate `agent.py` — it's not wired into the active platform setup.

---

## Issue 7: `LemonadeLLMAPI` Registration Timing

**File:** `__init__.py:31`

**Issue:** `llm.async_register_api(hass, LemonadeLLMAPI(hass))` is called in `async_setup()`, but the tools in `llm_tools.py` reference `DOMAIN` which is the same constant. This is correct, but the API registration happens BEFORE `async_setup_entry()` runs, meaning the API is available before any config entries exist.

When the conversation entity calls `chat_log.async_provide_llm_data()` with the API ID, HA looks up the registered API and calls `async_get_api_instance()`. If the config entry hasn't loaded yet, the tools may not be properly initialized.

**Impact:** Potential race condition at startup — if a conversation request arrives before the entry is fully loaded, tools may be empty.

**Fix:** Low priority. The `after_dependencies: ["intent"]` in manifest.json should handle ordering. But consider adding a guard in `async_get_api_instance()`.

## Proposed Fix Plan (Priority Order)

### Phase 1: Fix the Critical Streaming Tool Call Bug (Issue 1)
- **File:** `conversation.py` — `_async_handle_chat_log`
- After the `async for content in chat_log.async_add_delta_content_stream(...)` loop, detect accumulated tool calls in `chat_log` and execute them via `chat_log.async_call_tool()`
- Pattern: mirror the non-streaming fallback (lines 476-500)

### Phase 2: Fix Silent Exception Swallowing (Issue 4)
- **File:** `conversation.py` — lines 412, 424
- Add `_LOGGER.warning(...)` in all bare `except Exception` blocks

### Phase 3: Fix RAG API Key Issue (Issue 3)
- **File:** `__init__.py` — lines 96-99
- Pass `api_key` to `RAGIndex()` constructor, or remove the redundant creation

### Phase 4: Fix Non-Streaming tool_args Type (Issue 5)
- **File:** `conversation.py` — line 496
- Add type check before `json.loads()`

### Phase 5: Fix Schema Conversion (Issue 2)
- **File:** `conversation.py` — `_get_tools`
- Add `isinstance` check on `tool.parameters` and fallback for dict params

### Phase 6: Remove Dead Code (Issue 6)
- **File:** `agent.py`
- Either remove entirely or add clear deprecation notice
