# Plan: AnГЎlisis, CorrecciГіn de Bugs y Mejoras para Lemonade Conversation Advanced

## Resumen Ejecutivo

Este plan cubre: (A) bugs crГӯticos y errores encontrados en el cГіdigo, (B) mejoras arquitectГіnicas alineadas con los patrones modernos de HA, (C) un menГә de configuraciГіn renovado, y (D) nuevas funcionalidades de monitoreo/control via Lemonade Server API.

---

## Parte A: Bugs y Errores Encontrados

### A1. ARCHIVO MUERTO: `agent.py` вҖ” Clase duplicada no utilizada
- **Archivo**: `agent.py` (551 lГӯneas)
- **Problema**: Define `LemonadeConversationEntity(ConversationEntity)` que NUNCA se usa. La integraciГіn real usa `conversation.py` вҶ’ `LemonadeConversationEntity(ConversationEntity, LemonadeBaseEntity)`. El archivo agent.py parece ser un archivo legacy de una versiГіn anterior.
- **Impacto**: CГіdigo muerto, confusiГіn, imports no utilizados.
- **AcciГіn**: Eliminar `agent.py` completamente o renombrar como `agent_legacy.py` si se quiere preservar como referencia.

### A2. IMPORTS ROTOS en `mcp_server.py`
- **Archivo**: `mcp_server.py:24-30`
- **Problema**: Importa `CONF_LMSTUDIO_URL`, `DEFAULT_LMSTUDIO_URL` que NO existen en `const.py`. Los nombres reales son `CONF_LEMONADE_URL`, `DEFAULT_LEMONADE_URL`.
- **Impacto**: La integraciГіn fallarГЎ al iniciar si se usa esta ruta de importaciГіn.
- **AcciГіn**: Corregir imports a `CONF_LEMONADE_URL` y `DEFAULT_LEMONADE_URL`.

### A3. NOMBRE DE DOMINIO INCORRECTO
- **Archivo**: `mcp_server.py` (varias lГӯneas)
- **Problema**: El archivo referencia `DOMAIN` de `const.py` que es `"lemonade_conversation_advanced"`, pero el mcp_server parece asumir que es un dominio diferente en algunos contextos.
- **AcciГіn**: Verificar que todos los usos de DOMAIN sean consistentes.

### A4. RAGIndex creado en cada peticiГіn (no reutilizado)
- **Archivo**: `conversation.py:442-465`
- **Problema**: Cada llamada a `_async_handle_chat_log` crea un NUEVO `RAGIndex` y lo carga desde disco, ignorando el `rag_index` compartido almacenado en `hass.data[DOMAIN]["rag_index"]` (configurado en `__init__.py:107`).
- **Impacto**: Lecturas de disco innecesarias en cada request, potencial corrupciГіn si dos requests escriben simultГЎneamente.
- **AcciГіn**: Reusar `hass.data[DOMAIN]["rag_index"]` en lugar de crear uno nuevo.

### A5. `max_iterations` hardcodeado a 5
- **Archivo**: `conversation.py:434`
- **Problema**: `max_iterations = 5` ignora por completo el valor configurado `DEFAULT_MAX_ITERATIONS = 10` en const.py y el valor del subentry.
- **AcciГіn**: Leer de `self.subentry.data` con fallback a const.py.

### A6. `datetime.now()` en ConversationHistory
- **Archivo**: `conversation_history.py:104`
- **Problema**: Usa `datetime.now()` en lugar de `dt_util.now()` de HA. Esto puede causar inconsistencias de timezone.
- **AcciГіn**: Usar `from homeassistant.util import dt as dt_util` y `dt_util.utcnow()`.

### A7. MCP Server crea `aiohttp.ClientSession` por request
- **Archivo**: `agent.py:1120,1189`
- **Problema**: `_get_mcp_tools` y `_call_mcp_tool` crean un `aiohttp.ClientSession()` nuevo por cada llamada. Esto es ineficiente y puede agotar conexiones.
- **AcciГіn**: Usar `async_get_clientsession(self.hass)` que HA reutiliza.

### A8. `LemonadeBaseEntity` no hereda de HA base class
- **Archivo**: `entity.py:13`
- **Problema**: `LemonadeBaseEntity` es una clase mixin sin heredar de `Entity` o similar. Funciona pero no es idiomГЎtico.
- **AcciГіn**: Considerar heredar de `homeassistant.helpers.entity.Entity` o mantener como mixin pero documentar la razГіn.

### A9. Strings de traducciones desalineados con config_flow
- **Archivo**: `translations/en.json` y `strings.json`
- **Problema**: Los strings definen campos como `default_model`, `top_p`, `top_k`, `streaming`, `prompt` pero el config_flow real usa `model_name`, `system_prompt`, `temperature`, `max_tokens`, `llm_hass_api`, `enable_rag`, `rag_top_k`. Hay desfase completo.
- **AcciГіn**: Actualizar strings.json y translations/en.json para coincidir con el config_flow real.

### A10. `services.yaml` referencian servicios no implementados
- **Archivo**: `services.yaml`
- **Problema**: Define servicios como `load_model`, `unload_model`, `pull_model`, `list_models`, `get_system_info`, `get_stats`, `set_default_model`, `query_image`, `update_config` pero NO hay archivo `services.py` que los implemente. Los servicios del README no coinciden con el cГіdigo actual.
- **AcciГіn**: Crear `services.py` con los handlers, o eliminar las definiciones no implementadas.

### A11. F-strings en logging
- **Archivo**: `agent.py` (mГәltiples lГӯneas, ej: 179, 180, 560, 684, etc.)
- **Problema**: Usa `f"..."` en `_LOGGER.debug/info` en lugar de `%s` formatting. Esto evalГәa el string aunque el log level estГ© deshabilitado.
- **AcciГіn**: Cambiar a `_LOGGER.debug("Server Type: %s", self.server_type)` pattern.

### A12. Variable `_current_conversation_id` no inicializada
- **Archivo**: `agent.py:692`
- **Problema**: `self._current_conversation_id = conversation_id` se establece pero nunca se usa en la clase. Es dead code.
- **AcciГіn**: Eliminar la referencia.

## Parte B: Mejoras ArquitectГіnicas

### B1. Unificar las dos clases ConversationEntity
- **Problema**: `agent.py` y `conversation.py` ambas definen `LemonadeConversationEntity`. La de `conversation.py` es la que se usa (hereda de `LemonadeBaseEntity`). La de `agent.py` es legacy.
- **AcciГіn**: Eliminar `agent.py` completamente. Mover lГіgica Гәtil (si existe) a `conversation.py`.

### B2. Implementar servicios HA (`services.py`)
- **Problema**: Los servicios estГЎn definidos en `services.yaml` pero no implementados.
- **AcciГіn**: Crear `services.py` con handlers para:
  - `load_model` вҶ’ `POST /v1/load`
  - `unload_model` вҶ’ `POST /v1/unload`
  - `pull_model` вҶ’ `POST /v1/pull`
  - `list_models` вҶ’ `GET /v1/models`
  - `get_system_info` вҶ’ `GET /v1/system-info`
  - `get_stats` вҶ’ `GET /v1/stats`
  - `set_default_model` вҶ’ Actualizar config entry
  - `query_image` вҶ’ Vision API
  - `update_config` вҶ’ Actualizar config entry

### B3. Sensor Platform para Lemonade Server
- **Problema**: El README menciona 7 sensors pero no existe `sensor.py` en el cГіdigo actual.
- **AcciГіn**: Crear `sensor.py` con `DataUpdateCoordinator` que polleda:
  - `GET /v1/health` вҶ’ health status, loaded models, recipe info
  - `GET /v1/stats` вҶ’ TTFT, tokens/s, input/output tokens
  - `GET /v1/system-stats` вҶ’ CPU%, GPU%, VRAM, NPU%
  - `GET /v1/system-info` вҶ’ hardware details

### B4. Usar `async_add_delta_content_stream` correctamente
- **Problema**: El streaming actual lee `response.text()` completo antes de parsear. Para streaming real, deberГӯa usar `iter_lines()` o `iter_chunked()`.
- **AcciГіn**: Refactorizar `_iter_sse_deltas` para usar streaming real con `iter_lines()`.

### B5. AГұadir `reconfigure` step al config flow
- **PatrГіn moderno de HA**: El config flow actual no tiene `async_step_reconfigure` para cambiar la URL del servidor.
- **AcciГіn**: AГұadir step de reconfiguraciГіn para cambiar server_url y api_key sin recrear la entrada.
