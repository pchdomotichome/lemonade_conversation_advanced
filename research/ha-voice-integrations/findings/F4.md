# F4: HA Config Flow Patterns

## Findings

### [1] HA config flows are built on the Data Entry Flow framework, with ConfigFlow subclasses defining steps via `async_step_<step_id>` methods
- quote: "Data Entry Flow is a data entry framework that is part of Home Assistant. Data entry is done via data entry flows. A flow can represent a simple login form or a multi-step setup wizard for a component."
- url: https://developers.home-assistant.io/docs/data_entry_flow_index
- source_type: primary
- published: unknown
- confidence: high

### [2] Reserved step names in HA config flows: `user`, `reconfigure`, `reauth`, `import`, plus discovery-specific steps (`bluetooth`, `dhcp`, `hassio`, `homekit`, `mqtt`, `ssdp`, `usb`, `zeroconf`)
- quote: "There are a few step names reserved for system use: bluetooth, discovery (DEPRECATED), dhcp, hassio, homekit, mqtt, ssdp, usb, user, reconfigure, zeroconf, reauth, import."
- url: https://developers.home-assistant.io/docs/config_entries_config_flow_handler
- source_type: primary
- published: unknown
- confidence: high

### [3] Modern HA options flows support `OptionsFlowWithReload` subclass that auto-reloads the integration on option changes, replacing the older manual update_listener pattern
- quote: "If the integration should be reloaded after the config options change, it can subclass from `OptionsFlowWithReload` instead of `OptionsFlow`. `OptionsFlowWithReload` will automatically reload the integration once the options change."
- url: https://developers.home-assistant.io/docs/core/integration/options_flow
- source_type: primary
- published: unknown
- confidence: high

### [4] Config entry migration uses `VERSION` and `MINOR_VERSION` class attributes on ConfigFlow; minor version bumps are backwards-compatible while major bumps require `async_migrate_entry`
- quote: "If minor versions differ but major versions are the same, integration setup will be allowed to continue even if the integration does not implement `async_migrate_entry`. This means a minor version bump is backwards compatible unlike a major version bump."
- url: https://developers.home-assistant.io/docs/config_entries_config_flow_handler
- source_type: primary
- published: unknown
- confidence: high

### [5] The `reconfigure` step is the modern pattern for allowing users to change config entry data (e.g., host/port) without needing OptionsFlow, distinct from `reauth` which handles authentication issues
- quote: "A config entry can allow reconfiguration by adding a `reconfigure` step. This provides a way for integrations to allow users to change config entry data without the need to implement an `OptionsFlow` for changing setup data which is not meant to be optional."
- url: https://developers.home-assistant.io/docs/config_entries_config_flow_handler
- source_type: primary
- published: unknown
- confidence: high

### [6] Translation strings are structured in strings.json under `config`, `options`, and `config_subentries` keys, with `config_subentries` being a map-of-maps keyed by subentry type
- quote: "The translation strings for the configuration flow handler, the option flow handler and config subentry handlers are defined under the `config`, `options` and `config_subentries` keys respectively. Note that `config_subentries` is a map of maps, where the keys are the subentry types supported by the integration."
- url: https://developers.home-assistant.io/docs/internationalization/core
- source_type: primary
- published: unknown
- confidence: high

### [7] Translation strings support placeholder references via `[%key:component::domain::path::to::key%]` syntax, enabling reuse of common translations across integrations
- quote: "Strings which are used more than once should be not be duplicated, instead references should be used to refer to the single definition. The reference can be any valid translation key."
- url: https://developers.home-assistant.io/docs/internationalization/core
- source_type: primary
- published: unknown
- confidence: high

### [8] Translation strings support `flow_title` for dynamic flow titles, `progress` for async_show_progress actions, and `create_entry` for success dialog messages
- quote: "flow_title: Optional. Title to show in list. Only will be rendered if placeholders required... progress: This message will be displayed if `slow_task` is returned as `progress_action` for `async_show_progress`... create_entry: This message will be displayed in the success dialog if `async_create_entry` is called."
- url: https://developers.home-assistant.io/docs/internationalization/core
- source_type: primary
- published: unknown
- confidence: high

### [9] The MQTT integration (core) demonstrates ConfigSubentryFlow pattern: it imports `ConfigSubentryFlow` and `SubentryFlowResult` from `homeassistant.config_entries`, defining per-device-entity subentry flows for 21+ platform types
- quote: "from homeassistant.config_entries import (ConfigEntry, ConfigFlow, ConfigFlowResult, ConfigSubentryFlow, OptionsFlow, SubentryFlowResult,)"
- url: https://github.com/home-assistant/core/blob/dev/home-assistant/components/mqtt/config_flow.py
- source_type: primary
- published: 2024-2025
- confidence: high

### [10] HACS integration config flow (real-world HACS example) uses GitHub Device OAuth flow with `async_show_progress` for device activation, demonstrates `VERSION = 1` and combined ConfigFlow + OptionsFlow in single file with `async_get_options_flow` static callback
- quote: "class HacsFlowHandler(ConfigFlow, domain=DOMAIN): VERSION = 1... return self.async_show_progress(step_id='device', progress_action='wait_for_device', description_placeholders={'url': OAUTH_USER_LOGIN, 'code': self._registration.user_code}, progress_task=self.activation_task)"
- url: https://github.com/hacs/integration/blob/main/custom_components/hacs/config_flow.py
- source_type: primary
- published: 2024-2025
- confidence: high

### [11] Modern HA config flows support `add_suggested_values_to_schema()` for pre-populating form fields from existing config entry data, critical for options flows and reconfigure flows
- quote: "return self.async_show_form(step_id='init', data_schema=self.add_suggested_values_to_schema(OPTIONS_SCHEMA, self.config_entry.options))"
- url: https://developers.home-assistant.io/docs/core/integration/options_flow
- source_type: primary
- published: unknown
- confidence: high

### [12] The `async_show_progress` pattern with `progress_task` enables long-running async operations (like OAuth device activation) to show a progress UI while waiting, with automatic flow re-entry on completion
- quote: "show_progress_kwargs = { 'step_id': 'device', 'progress_action': 'wait_for_device', 'description_placeholders': {...}, 'progress_task': self.activation_task, } return self.async_show_progress(**show_progress_kwargs)"
- url: https://github.com/hacs/integration/blob/main/custom_components/hacs/config_flow.py
- source_type: primary
- published: 2024-2025
- confidence: high
